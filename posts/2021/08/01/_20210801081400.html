# SOLID原則

# 目的

SOLID原則についてちゃんと理解する為に、自分の言葉で説明する。

# 背景

なんとなくで理解しているような言葉を自分の言葉で説明できるようにしておいた方が良いと思ったので。
ググって出てくるような言葉でしか説明できないのであれば、他人が理解できるような説明はできない。

# SOLID原則とは

# Single responsibility principle

単一責任の原則

# 原文

A class should have only one reason to change（クラスが変更される理由は1つだけであるべき）

# 解釈

1つのクラスは1つの責任（＝役割）しか持たないようにする。
これはクラスに限った話ではなく、モジュールやメソッドも同様である。

# 目的

可読性、保守性を高くする為。

複数の責任を持っている場合、ある機能のために入れた変更が他の機能にも影響を与えてしまう危険性が高くなる。
責任を1つに絞ることにより意図せぬ影響を防ぎ、変更に強くなる。

他にも

  • 複数の責任を持っているときよりも処理がシンプルになる
  • クラスやメソッドの名前からどんな役割を持っているか推測しやすくなる

といったメリットがある。

# Open/closed principle

開放/閉鎖原則

# 解釈

よく見られる説明は

拡張に対して開いて (open) いなければならず、
修正に対して閉じて (closed) いなければならない

であるが、これを

拡張に対して開いている　＝　拡張できる 修正に対して閉じている　＝　修正できない（言い換えると、修正されない）

と解釈した。
これを自分の言葉で組み立てると

機能追加の際に拡張が可能であり、既存のコードは修正されないような実装にしよう

となった。

これを守ることで、既存のソースコードを変更することなく機能を追加することができるようになる。

具体的な方法としてはインターフェースを継承することである。
実装はインターフェースを継承したクラスで行う。
機能追加時はインターフェースや既存コードを修正することなく、インターフェースを継承した新たなクラスを追加することで実現することができる。

# 目的

機能の追加により、既存の機能やコードが影響を受けないようにする

# 備考

  • 提唱された当時と現在では意味合いが少し異なっているらしい

    • 開放/閉鎖原則 - Wikipedia (opens new window)

  • 機能の変更や修正について触れられていることも多いが、この原則は機能追加にだけ適用されるものではないかと考えている

    • 機能の変更や修正の場合、元の処理を別の処理に変更（置き換える）ので既存コードの修正が必要だと思う
    • そもそもの解釈が間違っていた場合はこの主張は消し飛ぶことになる

# 解釈
